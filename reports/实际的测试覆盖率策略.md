# 实际的测试覆盖率策略

## 🤔 为什么不需要 70% 的覆盖率？

你说得对！**不是所有代码都需要测试**。盲目追求高覆盖率是浪费时间。

### 不需要测试的代码

#### 1. **简单的 POJO / DTO**
```java
// ❌ 不需要测试
public class User {
    private String name;
    private int age;
    
    // getter/setter
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}
```
**原因**：没有逻辑，只是数据容器。

#### 2. **简单的配置类**
```java
// ❌ 不需要测试
@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();
    }
}
```
**原因**：只是配置，没有业务逻辑。

#### 3. **私有辅助方法（通过公共方法间接测试）**
```java
public class Calculator {
    // ✅ 需要测试
    public int calculate(int a, int b) {
        return add(a, b);
    }
    
    // ❌ 不需要单独测试（通过 calculate 间接测试）
    private int add(int a, int b) {
        return a + b;
    }
}
```

#### 4. **框架生成的代码**
- MyBatis Mapper 接口
- JPA Repository 接口
- Lombok 生成的代码

#### 5. **简单的委托方法**
```java
// ❌ 不需要测试
public void save(User user) {
    userRepository.save(user);  // 只是委托
}
```

## ✅ 需要测试的代码

### 1. **核心业务逻辑**
```java
// ✅ 必须测试
public class OrderService {
    public BigDecimal calculateTotal(Order order) {
        BigDecimal total = order.getSubtotal();
        if (order.hasDiscount()) {
            total = total.multiply(order.getDiscountRate());
        }
        total = total.add(order.getShippingFee());
        return total;
    }
}
```
**原因**：复杂的计算逻辑，容易出错。

### 2. **工具类和转换器**
```java
// ✅ 必须测试
public class ColumnNameCamel {
    public String toFieldName(String dbColumnName) {
        // 复杂的字符串转换逻辑
    }
}
```
**原因**：被多处使用，影响范围大。

### 3. **有条件分支的代码**
```java
// ✅ 必须测试
public String getStatus(int code) {
    if (code == 0) return "SUCCESS";
    else if (code == 1) return "PENDING";
    else if (code == 2) return "FAILED";
    else return "UNKNOWN";
}
```
**原因**：需要测试所有分支。

### 4. **边界条件处理**
```java
// ✅ 必须测试
public int divide(int a, int b) {
    if (b == 0) throw new ValidateException("除数不能为0");
    return a / b;
}
```
**原因**：边界条件容易被忽略。

### 5. **公共 API**
```java
// ✅ 必须测试
public interface DbService {
    void insert(Object entity);
    void update(Object entity);
    void delete(Object entity);
}
```
**原因**：对外暴露的接口，需要保证稳定性。

## 🎯 DLZ-DB 项目的实际测试策略

### 优先级 1：核心工具类（必须测试）

| 类 | 当前覆盖率 | 目标覆盖率 | 原因 |
|----|-----------|-----------|------|
| `ColumnNameCamel` | 89% | **90%+** | 字符串转换，被广泛使用 |
| `DbConvertUtil` | 5% | **80%+** | 数据转换，核心功能 |
| `KeyUtil` | ~100% | **100%** | 键名生成，简单但重要 |
| `DbTypeEnum` | 10% | **50%+** | 枚举，测试主要方法即可 |

### 优先级 2：业务逻辑类（选择性测试）

| 类 | 是否测试 | 原因 |
|----|---------|------|
| `Wrapper` 系列 | ⚠️ 部分测试 | 复杂的 SQL 构建逻辑，但需要数据库 |
| `SqlHelper` | ✅ 需要测试 | SQL 生成逻辑 |
| `BeanInfoHolder` | ⚠️ 部分测试 | 缓存逻辑，可以测试核心方法 |

### 优先级 3：不需要测试

| 类型 | 示例 | 原因 |
|------|------|------|
| **POJO** | `TableInfo`, `ColumnInfo` | 只有 getter/setter |
| **注解** | `@TableName`, `@TableId` | 框架使用，不需要测试 |
| **配置类** | `DataSourceConfig` | 只是配置 |
| **接口** | `IChained`, `ICondAddByFn` | 只是定义，实现类测试 |

**说明**：
- **短期目标**（1-2周）：测试核心工具类

## 💡 测试的真正目的

### ❌ 错误的目标
- 追求 70% 覆盖率数字
- 测试所有代码
- 为了测试而测试

### ✅ 正确的目标
- **防止回归**：确保修改不会破坏现有功能
- **文档作用**：测试即文档，展示如何使用
- **设计反馈**：难以测试的代码通常设计不好
- **信心保证**：重构时有测试保护

## 🚀 实用建议

### 1. 不要追求覆盖率数字

```
❌ 错误：我们必须达到 70% 覆盖率
✅ 正确：我们要测试核心功能和容易出错的地方
```

### 2. 测试要有针对性

```
❌ 错误：测试所有 getter/setter
✅ 正确：测试复杂的业务逻辑和边界条件
```

### 3. 测试要易于维护

```
❌ 错误：写 1000 行的测试代码测试 100 行的业务代码
✅ 正确：用简洁的测试覆盖关键场景
```

### 4. 测试要快速执行

```
❌ 错误：每次测试都启动 Spring 和数据库
✅ 正确：单元测试不依赖外部资源
```

## 🎯 总结

### 关键原则

1. **不是所有代码都需要测试**
2. **测试要有针对性，不追求数字**
3. **优先测试核心功能和容易出错的地方**
4. **测试要易于维护和快速执行**

### DLZ-DB 的实际目标

| 时间 | 覆盖率目标 | 重点 |
|------|-----------|------|
| **现在** | 2% | 已有核心工具类测试 ✅ |
| **1个月** | 15-20% | 完善核心类测试 |
| **3个月** | 30-40% | 测试主要业务逻辑 |
| **长期** | 30-50% | 保持稳定，不追求更高 |
